<Task title="Étape 2 : Création du service">

Créez un dossier `src/services` et à l'intérieur, un fichier `battleship.service.ts` :

```typescript title="src/services/battleship.service.ts"
import axios from 'axios';
import { type Ship } from '../types';

const axiosInstance = axios.create({
  baseURL: "http://www.backend.url",
});

export default class BattleshipService {
  static getShips(): Promise<Ship[]> {
    return axiosInstance
      .get('/api/ships')
      .then((resp) => resp.data)
      .catch((e) => Promise.reject(e));
  }
}
```

Ce service contient une méthode statique `getShips()` qui effectue une requête HTTP GET vers l'API pour récupérer la liste des navires.

C'est dans ce fichier `battleship.service.ts` que vous ajouterez les nouvelles routes backend pour la suite du TP.

</Task>

<div id="import-export" class="tp-anchor" />

<Aside type="note" title="Comprendre `import` et `export`">
Le service que vous venez d'écrire utilise les **deux variantes d'export** du JavaScript moderne. Il est important de bien les distinguer.

**Export *default*** — un seul par fichier, généralement le composant ou la classe principale. À l'import, le nom est **libre**.

```ts title="battleship.service.ts"
// la classe est exportée comme « valeur par défaut » du fichier
export default class BattleshipService { ... }
```

```ts title="ShipsPage.tsx"
// le nom est libre — il pourrait s'appeler Service, Api, etc.
import BattleshipService from '../services/battleship.service';
```

**Export *nommé*** — plusieurs par fichier, typique pour les types et les utilitaires. À l'import, on encadre les noms entre **accolades**, et ils doivent **correspondre exactement** à ceux exportés.

```ts title="types.ts"
export interface Ship { ... }
export type CellState = 'empty' | 'ship' | 'hit';
```

```ts title="ShipsPage.tsx"
// noms exacts entre accolades
import { type Ship, type CellState } from '../types';
```

Les deux peuvent se combiner dans un même `import` :

```ts
import React, { useState, useEffect } from 'react';
//     ↑ default        ↑ exports nommés
```
</Aside>

## Découverte de `useState`

Avant de charger des données depuis le serveur, familiarisons-nous avec `useState`, le hook fondamental de React pour gérer l'état local d'un composant.

<Aside type="note" title="Qu'est-ce qu'un état (state) ?">
L'état d'un composant, c'est une donnée qui peut **changer au cours du temps** et qui provoque un **re-rendu** du composant quand elle change. Un compteur de clics, la valeur d'un champ de texte, un booléen `isLoading` : ce sont tous des états.

`useState` retourne un tableau de deux éléments :
- la valeur actuelle de l'état ;
- une fonction setter pour la modifier.
</Aside>

<Task title="Exercice : un compteur">

Créez un composant `Counter.tsx` qui affiche un compteur et deux boutons :

```tsx title="src/components/Counter.tsx"
import { useState } from 'react';

const Counter = () => {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Compteur : {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
    </div>
  );
};

export default Counter;
```

Ajoutez ce composant temporairement dans votre application et observez son comportement. Essayez de l'utiliser **deux fois** sur la même page : chaque instance possède son propre état indépendant.

</Task>

<Task title="Exercice : un champ contrôlé">

En React, un champ de formulaire est dit **contrôlé** lorsque sa valeur est pilotée par un état. Créez un composant qui affiche en temps réel ce que vous tapez :

```tsx title="src/components/ControlledInput.tsx"
import { useState } from 'react';

const ControlledInput = () => {
  const [value, setValue] = useState('');

  return (
    <div>
      <input
        type="text"
        value={value}
        onChange={(e) => setValue(e.target.value)}
      />
      <p>Vous avez tapé : {value}</p>
    </div>
  );
};

export default ControlledInput;
```

Ce pattern (valeur liée à l'état + `onChange` qui appelle le setter) est la base de tous les formulaires React. Vous le retrouverez dans la section [Authentification](/authentification).

</Task>

### Récupérer les navires et les stocker dans un état

<div id="useeffect" class="tp-anchor" />
<div id="affichage-conditionnel" class="tp-anchor" />

<Task title="Créer le composant de page">

Sur votre nouvelle page, créez un composant qui :
- utilise le hook **`useState`** pour stocker la liste des navires ;
- utilise le hook **`useEffect`** pour déclencher l'appel API **une seule fois** au montage du composant — placer l'appel hors de `useEffect` le ferait rejouer à chaque rendu et provoquerait une **boucle infinie de requêtes** ;
- utilise un composant dédié pour afficher chaque navire.

Vous trouverez plus de détails sur [useState](https://react.dev/reference/react/useState) et [useEffect](https://react.dev/reference/react/useEffect) dans la documentation officielle React.

</Task>

<Aside type="caution" title="Pourquoi `useEffect` est indispensable ici">
Le corps d'une fonction-composant est **ré-exécuté à chaque rendu**. Si vous appelez `BattleshipService.getShips()` directement dans le corps du composant :

1. l'appel API se déclenche ;
2. le `.then()` exécute `setShips(data)` ;
3. ce setter provoque un **nouveau rendu** ;
4. qui ré-exécute le corps… et **relance l'appel API**.

→ Boucle infinie de requêtes. `useEffect(() => { ... }, [])` isole l'appel et garantit qu'il ne s'exécute qu'**une seule fois**, après le premier rendu.
</Aside>

<Example title="Exemple de mise en œuvre">

Voici un exemple complet pour la page qui récupère et affiche les navires :

```tsx title="src/pages/ShipsPage.tsx"
import React, { useEffect, useState } from 'react';
import BattleshipService from '../services/battleship.service';
import { type Ship } from '../types';
import { Box, Heading, Text, Spinner } from '@chakra-ui/react';

const ShipsPage: React.FC = () => {
  const [ships, setShips] = useState<Ship[]>([]);
  const [isLoading, setIsLoading] = useState<boolean>(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    BattleshipService.getShips()
      .then(data => {
        setShips(data);
        setIsLoading(false);
      })
      .catch(err => {
        console.error('Erreur lors de la récupération des navires:', err);
        setError('Impossible de charger la liste des navires');
        setIsLoading(false);
      });
  }, []); // tableau vide = s'exécute une seule fois, au premier rendu

  if (isLoading) {
    return (
      <Box textAlign="center" p={8}>
        <Spinner size="xl" />
        <Text mt={4}>Chargement des navires...</Text>
      </Box>
    );
  }

  if (error) {
    return <p>{error}</p>;
  }

  return (
    <Box p={5}>
      <Heading as="h1" mb={6}>Liste des navires</Heading>
      {JSON.stringify(ships)}
    </Box>
  );
};

export default ShipsPage;
```

Ce composant montre comment :
- déclarer les états avec `useState` pour gérer les données, le chargement et les erreurs ;
- utiliser `useEffect` pour déclencher l'appel API au chargement du composant ;
- afficher différents contenus selon l'état (chargement, erreur ou données).

Il vous reste à créer le composant `ShipCard` qui affichera chaque navire, et à l'utiliser dans le `return`.

</Example>

<Aside type="note" title="Comprendre le tableau de dépendances de useEffect">
Le deuxième argument de `useEffect` est le **tableau de dépendances**. Il contrôle *quand* l'effet s'exécute :

- **`[]`** (tableau vide) : l'effet s'exécute *une seule fois*, après le premier rendu — c'est le bon choix pour charger des données initiales ;
- **`[variable]`** : l'effet se re-déclenche chaque fois que `variable` change — utile pour recharger des données quand un filtre évolue ;
- **sans tableau** : l'effet s'exécute après *chaque* rendu — rarement voulu, souvent source de boucle infinie.
</Aside>

<Aside type="note">
Dans la suite du TP, nous utiliserons une librairie externe pour stocker les données récupérées sur le serveur.
</Aside>

<Aside type="caution" title="Erreur CORS (Cross-Origin Request)">
Si vous rencontrez une erreur **Blocage d'une requête multiorigines (Cross-Origin Request)**, ajoutez la ligne suivante dans `Application::execute()` juste avant `$router->dispatch()` :

```php

```

## Affichage des navires

<Aside type="note" title="Props vs State">
Deux concepts fondamentaux de React :

- **State (état)** : données appartenant *au composant lui-même*, déclarées avec `useState`. Quand l'état change, React re-rend le composant.
- **Props (propriétés)** : données reçues *depuis le composant parent*. Un composant enfant ne peut pas modifier ses props — il les lit seulement.

**Règle pratique** : si la donnée peut changer et appartient à ce composant, c'est du **state**. Si elle vient d'un parent, c'est une **prop**.
</Aside>

### Étape 2 : Passer les données via les props

<div id="props" class="tp-anchor" />

<Task title="Étape 2 : Définir les props et injecter les données">

Faisons en sorte que le composant reçoive les données depuis son parent. On définit une interface TypeScript pour documenter ce que le composant attend :

```tsx title="src/components/ShipCard.tsx"
import { type Ship } from '../types';

interface ShipCardProps {
  ship: Ship;
}

const ShipCard = ({ ship }: ShipCardProps) => {
  return (
    <div>
      <h2>{ship.name}</h2>
      <p>Taille : {ship.length} cases</p>
    </div>
  );
};

export default ShipCard;
```

</Task>

<Aside type="note" title="Lire les props avec la destructuration">
La signature `({ ship }: ShipCardProps)` utilise la **destructuration** d'objet, un raccourci ES6 pour extraire des champs d'un objet.

Trois écritures strictement équivalentes :

```tsx
// 1. Sans destructuration — on accède via props.ship
const ShipCard = (props: ShipCardProps) => {
  return <h2>{props.ship.name}</h2>;
};

// 2. Destructuration en début de fonction
const ShipCard = (props: ShipCardProps) => {
  const { ship } = props;
  return <h2>{ship.name}</h2>;
};

// 3. Destructuration directe dans la signature — la forme idiomatique
const ShipCard = ({ ship }: ShipCardProps) => {
  return <h2>{ship.name}</h2>;
};
```

C'est la **3ᵉ forme** qu'on utilise en React. Elle s'étend naturellement à plusieurs props : `({ ship, isPlaced, onClick }: ShipCardProps)`.
</Aside>

### Étape 3 : Afficher la liste avec `.map()`

<Task title="Étape 3 : Utiliser le composant dans la liste">

Dans `ShipsPage.tsx`, remplacez le `JSON.stringify(ships)` par un rendu de la liste :

```tsx
return (
  <Box p={5}>
    <Heading as="h1" mb={6}>Liste des navires</Heading>
    {ships.map((ship) => (
      <ShipCard key={ship.key} ship={ship} />
    ))}
  </Box>
);
```

</Task>

<Aside type="caution" title="La prop key est obligatoire dans un .map()">
Elle permet à React d'identifier chaque élément de façon unique pour optimiser les re-rendus. Évitez d'utiliser l'index du tableau comme clé : préférez un identifiant métier stable comme `ship.key`.
</Aside>

<div id="immutabilite" class="tp-anchor" />
<div id="es6-spread" class="tp-anchor" />

<Aside type="caution" title="Ne jamais muter le state directement">
React ne détecte les changements que via la fonction setter, en comparant la **référence** de la nouvelle valeur à l'ancienne. `push` modifie le tableau existant : la référence reste la même, React ne voit rien.

```tsx
// ❌ INCORRECT — React ne verra pas le changement
ships.push(newShip);

// ✅ CORRECT — on passe un nouveau tableau
setShips([...ships, newShip]);
```
</Aside>

<Aside type="note" title="Le spread operator (`...`) en détail">
Le `...` est l'**opérateur de décomposition** d'ES6. Il *« déballe »* les éléments d'un tableau ou les champs d'un objet à l'endroit où il est utilisé. Trois usages courants :

```ts
// 1. Copier un tableau et ajouter un élément
const ships = ['Destroyer', 'Carrier'];
const next = [...ships, 'Submarine'];
// → ['Destroyer', 'Carrier', 'Submarine']

// 2. Copier un objet et modifier un champ
const ship = { name: 'Destroyer', length: 3 };
const updated = { ...ship, length: 5 };
// → { name: 'Destroyer', length: 5 }

// 3. Fusionner deux objets
const merged = { ...defaults, ...userPrefs };
```

En React, c'est cet opérateur qui permet de respecter l'**immutabilité** : on ne touche jamais à l'objet ou au tableau original, on en crée un nouveau qui en hérite.
</Aside>

<div id="re-render" class="tp-anchor" />

<Aside type="note" title="Quand un composant se re-rend-il ?">
Un composant React est re-rendu dans **trois** cas :

1. **Son state change** (via un setter `useState`) ;
2. **Ses props changent** (le parent lui passe une nouvelle valeur) ;
3. **Son parent se re-rend** (par défaut, tous les enfants sont aussi re-rendus).

C'est pour cette raison que l'immutabilité est essentielle : sans nouvelle référence, React ne déclenche pas le re-render.
</Aside>
